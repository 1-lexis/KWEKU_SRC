Section 2

Accessibility Testing

The website was evaluated for WCAG 2.1 compliance, screen reader compatibility, and general usability for persons with disabilities.

Screen Reader Testing

Tool Used

NVDA (NonVisual Desktop Access) — free, open-source screen reader for Windows — was used to navigate all pages with keyboard-only input.

Accessibility Features Implemented

  • ARIA labels: All interactive elements (buttons, links, form inputs, navigation regions) include aria-label or aria-labelledby attributes so screen readers announce them correctly.
  • ARIA roles: role="navigation", role="main", role="search", role="contentinfo", role="tabpanel", and role="listbox" are used throughout to provide semantic structure.
  • Keyboard navigation: All interactive elements (nav links, buttons, time slots, tabs, accordion) are reachable and operable using Tab, Enter, and Space keys. Skip-to-content link is available on focus.
  • High-contrast compatibility: CSS custom properties use high-contrast gold-on-dark colour combinations exceeding WCAG AA ratio (4.5:1) for text and interactive elements.
  • Proper form labelling: Every form input has a corresponding <label> linked via for/id. Required fields use both the required attribute and aria-required="true". Error messages use role="alert".
  • Semantic HTML5: <header>, <nav>, <main>, <article>, <section>, <aside>, and <footer> are used throughout, giving screen readers a clear document outline.
  • Alt text / ARIA labels for decorative content: All decorative emoji and icons include aria-hidden="true" to prevent screen readers announcing them. Meaningful images include descriptive aria-label attributes.
  • Live regions: Dynamic content areas (search results, form alerts, no-results messages) use aria-live="polite" so screen readers announce updates without interrupting the user.

Issues & Improvements Made

  • Initial hamburger button had no accessible label — added aria-label="Toggle mobile menu" and aria-expanded state tracking.
  • Time slots in the booking form had no keyboard access — added tabindex="0", role="radio", and aria-checked attributes, plus keydown handlers for Enter and Space.
  • Accordion items initially had no state — added aria-expanded on each header button.

⚠️ Minor Issues

accent-color for checkboxes was not styled on Safari 15 and below. Added fallback. :has() pseudo-class not available in Safari below version 15.4 — added fallback JavaScript class toggle.

Inconsistencies & Causes

  • backdrop-filter (Firefox): Firefox requires a -webkit-backdrop-filter prefix alongside the standard property. This is a historical vendor implementation difference. Fixed with both properties in the CSS.
  • :has() pseudo-class (Safari <15.4): This CSS Level 4 selector was not available in older Safari builds. A JavaScript fallback toggles a class .sport-selected on checkbox change to replicate the styling effect.
  • Custom date/time inputs (all browsers): <input type="date"> renders differently across browsers. A consistent neutral style was applied using CSS to minimise visual differences while maintaining functionality.

Evaluation

Summary of Testing Process

The testing process for the SRC Sports Academy website was conducted systematically across four stages: HTML validation, CSS validation, accessibility evaluation, and cross-browser compatibility testing. Each stage revealed minor issues that were corrected before re-testing confirmed resolution. The iterative correction cycle was effective and ensured a clean, standards-compliant codebase.

HTML validation highlighted structural issues such as missing ARIA heading associations and duplicate IDs — both critical for accessibility and DOM integrity. CSS validation flagged vendor extension warnings, which were documented rather than removed, as they serve legitimate browser compatibility purposes. Accessibility testing using NVDA revealed missing keyboard interactions on custom components like time slots and accordion toggles, which were subsequently fixed to ensure full keyboard navigability.

Role of W3C in Web Standards Enforcement

The World Wide Web Consortium (W3C) is the primary international standards organisation for the web. Founded by Tim Berners-Lee in 1994, it develops and maintains specifications for HTML, CSS, accessibility guidelines (WCAG), and many other web technologies. By providing free validation tools, the W3C enables developers to verify that their code conforms to published specifications, ensuring interoperability across browsers, devices, and assistive technologies.

Compliance with W3C standards is not merely academic — it directly impacts a website's accessibility, search engine ranking, and cross-browser reliability. The WCAG 2.1 guidelines referenced throughout this project represent the W3C's most impactful contribution to inclusive web design, providing measurable success criteria (Levels A, AA, and AAA) against which any website can be objectively evaluated.

Outstanding Issues & Recommendations

  • Online payment integration is outstanding. Currently booking relies on email confirmation. The recommendation is to integrate a Ghanaian payment gateway such as Paystack or Hubtel in the next development phase.
  • Real backend / database — form submissions are currently front-end only (simulated). A PHP/Node.js or Firebase backend should be connected to persist registrations, bookings, and contact messages.
  • Authentication system — a proper login/logout system with session management needs to be built to support returning members.
  • Twitter/X feed widget — the optional embedded Twitter feed was not implemented as Twitter's embed policy now requires API keys. Recommendation: use a social aggregator widget in the next iteration.

Reflection on Improvements Made

The most impactful improvements made during testing were the accessibility enhancements — particularly the keyboard navigation fixes and ARIA attribute additions. These changes made the website usable by a significantly wider audience at minimal code cost. The cross-browser fixes for backdrop-filter and :has() were important lessons in the gap between modern CSS specification availability and real-world browser support, reinforcing the need to always test across multiple environments rather than relying on a single browser during development.

Additional Analysis

Web Development Tools Evaluated

1. Visual Studio Code (VS Code)

VS Code was the primary code editor used throughout this project. Its key strengths for this project included:

  • Live Server extension: Enabled real-time browser preview with automatic reload on file save, significantly speeding up the development and testing cycle.
  • HTML/CSS IntelliSense: Auto-completion for HTML tags, CSS properties, and attribute values reduced syntax errors and improved coding speed.
  • Prettier formatter: Automatically formatted code for consistent indentation and readability, important for a multi-file project.
  • Emmet abbreviations: Allowed rapid HTML scaffolding using shorthand syntax, e.g., ul>li*5 generates a 5-item unordered list instantly.

Evaluation: VS Code proved to be an exceptionally powerful tool for this project. Its extensive extension ecosystem, zero licensing cost, and cross-platform availability make it the industry-standard choice for web development. The one limitation encountered was that complex CSS variable refactoring required manual find-and-replace rather than semantic rename refactoring (which is better supported for JavaScript).

2. Chrome DevTools

Chrome DevTools was used extensively for debugging, responsive design testing, and performance analysis.

  • Device Toolbar (Responsive Mode): Allowed simulation of various screen sizes (320px phone, 768px tablet, 1024px desktop) to test media queries and layout breakpoints without physical devices.
  • Elements Panel: Real-time inspection and modification of the DOM and CSS enabled rapid visual debugging — for example, identifying why the hamburger menu was not aligning correctly on small screens.
  • Lighthouse Audit: Automated audit tool providing scores for Performance, Accessibility, Best Practices, and SEO. Initial accessibility score was 84/100; after ARIA improvements it improved to 97/100.
  • Console Panel: JavaScript errors (such as the initial null reference when search-input was queried before the nav injection) were caught and debugged here.

Evaluation: Chrome DevTools is indispensable for modern web development. Its Lighthouse integration makes it uniquely powerful as both a debugging tool and a quality assurance mechanism, covering accessibility, performance, and SEO in a single automated pass. The limitation is that it only reflects Chrome's rendering engine — results may differ in Firefox or Safari.

Three Web Services & Their Relevance

1. Google Fonts (fonts.googleapis.com)

Google Fonts is a free web font hosting service used to load Bebas Neue and DM Sans in this project. Relevance: web fonts enable designers to use consistent, brand-appropriate typography across all devices without requiring font installation. Google Fonts reduces load time through edge caching and serves fonts over HTTPS with strong CDN infrastructure. Its relevance to modern web development is significant — over 60% of websites use at least one Google Font. For the SRC Sports Academy, the distinctive font pairing elevates brand identity beyond generic system fonts.

2. jQuery (code.jquery.com CDN)

jQuery is a JavaScript library used in this project to power the on-page search and filter functionality. Loaded from the jQuery CDN, it provides cross-browser DOM manipulation, event handling, and animation utilities with a concise syntax. While modern vanilla JavaScript can replicate most jQuery functionality, jQuery remains relevant for projects where quick prototyping, browser compatibility, and readable DOM manipulation code are priorities. For this assignment, jQuery satisfies the specific requirement for "jQuery-based on-page search functionality."

3. YouTube Embed API (youtube.com/embed)

The YouTube embed service allows third-party websites to embed YouTube videos using an <iframe>. Used on the home page of this project to embed Olympic sports highlights and CAF football content. Relevance: video content significantly increases user engagement and dwell time. Embedding from YouTube means the Academy website benefits from YouTube's global CDN infrastructure, adaptive streaming, and closed caption support without any hosting cost. This is the standard approach for multimedia-rich informational websites and demonstrates the value of third-party web service integration in modern development.

Future Improvements & Recommendations

  • Scalability: Migrate static HTML to a CMS (e.g. WordPress, Contentful) or a JavaScript framework (React/Next.js) to allow non-technical staff to update content without editing HTML files directly.
  • Payment Integration: Integrate Paystack or Hubtel payment gateway to enable online fee collection for memberships and session bookings, replacing the current email-only system.
  • User Dashboards: Build authenticated member dashboards where students can view their bookings, download receipts, track attendance, and update their profiles.
  • Progressive Web App (PWA): Convert the website into a PWA with a service worker and manifest file, enabling offline access and mobile home-screen installation — important for students with intermittent internet access.
  • Performance optimisation: Implement image compression (WebP format), lazy loading for all below-the-fold images, and CSS/JS minification to improve page load speeds on lower-bandwidth connections common in Ghana.
  • Database backend: Connect a MySQL or Firebase database to store registrations, bookings, and contact messages persistently, replacing the current front-end-only simulation.
  • Analytics: Integrate Google Analytics 4 to replace the simple localStorage visitor counter with actionable data on user behaviour, popular pages, and conversion funnels.
